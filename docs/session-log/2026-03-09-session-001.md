## Research Phase 1: SMP Server Architecture

### How the Haskell Reference Server Works

The simplexmq Haskell server (AGPL-3.0) at github.com/simplex-chat/simplexmq is the only existing SMP server implementation. Key findings from analyzing the source code:

**Concurrency model:** Three threads per client connection - reader, writer, processor. The reader parses incoming 16KB blocks, the processor dispatches commands, the writer serializes responses. This maps directly to Go's goroutine model.

**Queue model:** Unidirectional simplex queues. Each queue has exactly one sender and one recipient, identified by random 24-byte IDs. A full duplex conversation uses two queues, potentially on different servers.

**Subscription semantics:** Only one connection can subscribe to a queue at a time. When a new connection subscribes, the old one receives an END notification. Messages stay in the queue until explicitly acknowledged via ACK.

**Server-side re-encryption:** The server re-encrypts messages before delivery using a DH-derived shared secret with msgId as nonce via NaCl crypto_box. This prevents traffic correlation even if TLS is compromised.

**Transport:** Fixed 16,384-byte padded blocks after TLS handshake. First 2 bytes are content length (uint16 big-endian), remaining bytes are content plus '#' padding. Multiple transmissions can be batched in a single block.

### What This Means for GoRelay

The Haskell server's architecture is clean and maps well to Go. The three-thread model becomes three goroutines. STM (Software Transactional Memory) for subscription management becomes sync.Map + channels. The queue store becomes BadgerDB with prefix-based key schema.

No existing Go implementation of SMP exists anywhere. GoRelay will be the first.

## Research Phase 2: Go Server Patterns

### Goroutine-per-Connection Architecture

Go's runtime uses epoll/kqueue internally. Blocking conn.Read() calls park the goroutine and free the OS thread. Each goroutine starts at approximately 2KB stack. At 3 goroutines per connection, 10,000 concurrent clients consume approximately 60MB of stack memory. The practical limit is file descriptors, not goroutines.

### TLS 1.3 with Custom ALPN

Go's crypto/tls natively supports custom ALPN and non-HTTP TLS servers. Configuration is straightforward:

```go
tlsConfig := &tls.Config{
    Certificates: []tls.Certificate{cert},
    MinVersion:   tls.VersionTLS13,
    NextProtos:   []string{"smp/1"},
}
listener, _ := tls.Listen("tcp", ":5223", tlsConfig)
```

### Fixed 16KB Block Transport

The block transport uses io.ReadFull for exact reads (critical - plain conn.Read may return partial data). Read buffer is a fixed array on the struct to avoid per-read allocation.

### Technology Stack Decision

| Component | Choice | Rationale |
|-----------|--------|-----------|
| Language | Go 1.24+ | Single binary, goroutines, stdlib crypto with ML-KEM |
| Persistence | BadgerDB | Pure Go, native TTL, write-optimized LSM, concurrent writers |
| Configuration | koanf | Lightweight, proper key handling (Viper lowercases keys) |
| Logging | log/slog | Stdlib since Go 1.21, zero external dependencies |
| Metrics | Prometheus client_golang | Industry standard, separate port from relay |
| Rate Limiting | golang.org/x/time/rate | Token bucket, per-IP and per-connection |

**Why BadgerDB over alternatives:**

| Metric | BBolt | BadgerDB | Pebble | SQLite |
|--------|-------|----------|--------|--------|
| Individual writes | ~340 ops/s | ~40K ops/s | ~3.2K ns/op | ~1-5K ops/s |
| Batch writes | 305K ops/s | 1.82M ops/s | 484K ops/s | ~100-200K ops/s |
| Native TTL | No | Yes | No | No |
| Concurrent writers | No | Yes | Yes | No |

BadgerDB wins on write throughput (100x over BBolt), native TTL for automatic message expiry, and concurrent writers for multiple sender goroutines.

## Research Phase 3: Security Improvements Over SMP

### The Group Chat Problem

Group chats are the security weakness of every encrypted messenger. SimpleX sends each group message individually to every member. Signal uses Sender Keys (weaker forward secrecy). Matrix uses Megolm (weaker forward secrecy). By eliminating groups entirely, every message in GoRelay is guaranteed 1:1 with full Double Ratchet. No compromises.

### Six Concrete Improvements

**1. Noise Protocol replaces TLS.** WireGuard's transport protocol. 35-page specification vs TLS 1.3's 160 pages. No Certificate Authorities, no cipher negotiation (eliminates downgrade attacks), built-in identity hiding. The server's 32-byte Curve25519 public key IS the fingerprint - no certificate chain needed.

**2. Mandatory post-quantum cryptography.** Hybrid X25519 + ML-KEM-768 key exchange. Go 1.24 includes FIPS-validated, Trail-of-Bits-audited ML-KEM in the standard library. One fixed cipher suite per protocol version, no negotiation, no fallback.

**3. Two-hop relay routing as default.** Every message traverses two servers. Server A sees sender IP but not destination. Server B sees destination but not sender IP. SimpleX has this optionally (PMR since v5.8); GoRelay makes it mandatory.

**4. Server-generated cover traffic.** Poisson-distributed dummy messages encrypted identically to real messages. A network observer cannot distinguish active chatting from an idle connection.

**5. Automatic queue rotation.** Queue IDs rotate every 24-72 hours with random jitter. Old tracking becomes worthless. SimpleX supports manual rotation; GoRelay automates it.

**6. Aggressive message retention.** Delete on delivery ACK. 48-hour default TTL for undelivered messages. 7-day hard maximum. No IP logging, no metadata on disk.

## Research Phase 4: Dual-Protocol Architecture

### The Bridge Problem - Solved

The key architectural insight: GoRelay IS the bridge between SMP and GRP. No separate bridge service needed.

**Port 5223** speaks SMP (TLS 1.3, ALPN "smp/1") for SimpleX compatibility.
**Port 7443** speaks GRP (Noise Protocol, post-quantum, cover traffic) for enhanced security.

Both frontends share the same internal core:
- QueueStore interface backed by BadgerDB
- SubscriptionHub with channel-based message routing
- Same queue records, same message format internally

A message pushed via SMP is stored in the same queue and delivered to a GRP subscriber (and vice versa). The transport layer is the only difference.

### Client-Side Protocol Selection

SimpleGo devices select the protocol based on the server URI scheme:
- `smp://fingerprint@host:5223` selects SMP mode
- `grp://fingerprint@host:7443` selects GRP mode

When communicating with a SimpleX app user, the SimpleGo device uses SMP. When communicating with another SimpleGo device through a GoRelay server, it uses GRP with all enhancements.

### Server Architecture Diagram


## Research Phase 5: Noise Protocol Deep Dive

### Handshake Patterns for GoRelay

**Noise_IK (primary):** 1 round trip, 0-RTT encrypted payloads. Client knows server's static key in advance. Client's static key is encrypted. This is WireGuard's pattern.

**Noise_XX (fallback):** 1.5 round trips. No prior key knowledge needed. Both static keys encrypted during handshake. Strongest identity hiding.

**Noise_NK:** Server-only authentication, anonymous clients. Useful for anonymous queue creation.

### Why Noise Beats TLS for a Relay Server

| Property | Noise | TLS 1.3 |
|----------|-------|---------|
| Handshake size | 32-144 bytes | 1-4 KB |
| Implementation size | ~1,500 lines | ~15,000+ lines |
| CA dependency | None | Requires PKI/X.509 |
| Identity hiding | Both parties encrypted | Server name visible via SNI |
| Cipher negotiation | Impossible (fixed suite) | Possible |

### Go Library: flynn/noise v1.1.0

BSD-3 licensed, supports all 12 interactive patterns, PSK modes, rekey, channel binding. Clean API, used by 165 packages.

For post-quantum extension: katzenpost/noise provides Hybrid Forward Secrecy with Kyber1024. AGPL-3.0 licensed.

## Research Phase 6: Post-Quantum Cryptography

### NIST Standardization Complete

FIPS 203 (ML-KEM), FIPS 204 (ML-DSA), and FIPS 205 (SLH-DSA) were all finalized on August 13, 2024. ML-KEM-768 is the primary key encapsulation standard.

### Go 1.24 Standard Library Support

The crypto/mlkem package added ML-KEM-768 and ML-KEM-1024 in Go 1.24 (February 2025). FIPS 140-3 validated (CAVP certificate A6650), audited by Trail of Bits. Pure Go, no CGo.

Key sizes for ML-KEM-768: encapsulation key 1,184 bytes, ciphertext 1,088 bytes, shared secret 32 bytes.

### GRP Fixed Cipher Suite

```
GRP/1 cipher suite (non-negotiable):
  Key Exchange:  X25519 + ML-KEM-768 hybrid
  KDF:           HKDF-SHA-256
  Signatures:    Ed25519 (upgrade path to ML-DSA-65)
  AEAD:          ChaCha20-Poly1305
  Hash:          BLAKE2s
```

No version negotiation. First protocol byte identifies version. Version 1 requires PQC with no fallback.

## Research Phase 7: Legal and Retention Analysis

### GDPR and German Law Favor Zero-Knowledge

GDPR Article 5(1)(c) (data minimization) and 5(1)(e) (storage limitation) mandate minimal data collection. A relay's purpose is delivery, not storage - immediate deletion on delivery is ideal.

German Vorratsdatenspeicherung (data retention) was suspended by BNetzA since 2017. The CJEU ruled on September 20, 2022 (SpaceNet/Telekom Deutschland) that Germany's provisions violate EU law.

If metadata is never generated, freeze orders have nothing to preserve.

## Decisions Made

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Server language | Go | Single binary, goroutines, stdlib PQC |
| Dual-protocol | SMP + GRP on separate ports | Compatibility + innovation |
| GRP transport | Noise Protocol (IK/XX) | No CA, no negotiation, identity hiding |
| PQC | X25519 + ML-KEM-768, mandatory | FIPS 203, Go stdlib, audited |
| Persistence | BadgerDB | Native TTL, write-optimized, pure Go |
| Groups | Eliminated | Full Double Ratchet for every message |
| Cover traffic | Poisson-distributed server dummies | Traffic analysis resistance |
| Relay routing | Mandatory two-hop | Sender-recipient unlinkability |
| Message TTL | 48h default, 7d hard max | GDPR compliance, minimal attack surface |
| Documentation | Docusaurus, separate research + spec | Publishable research, auditable spec |
| Session logging | Date-prefixed, same pattern as SimpleGo | Proven workflow |
