## Overview

Season 2 took GoRelay from a compiled skeleton with PING/PONG to a fully functional SMP v7 server that passes the official SimpleX Chat server test. Starting with `go mod tidy` and ending with a VPS deployment verified by SimpleX Desktop, every Phase 1 task was completed: TLS CA chain, version handshake, queue operations (NEW, KEY, SKEY, SUB, SEND, MSG, ACK, DEL), BadgerDB persistent storage, per-message AES-256-GCM encryption, cryptographic deletion, and an embedded admin dashboard.

The journey required solving 8 protocol compatibility issues against the real SimpleX Chat client, each on its own fix branch. The most significant discoveries: SMP requires '#' padding (not zero bytes), TLS 1.2 is required for tls-unique channel binding, error responses must be text strings not binary bytes, and IDS response format is version-dependent.

47+ tests passing. SimpleX Chat server test green. VPS deployed and operational.

## Phase 1 Implementation (9 Tasks)

### Task 1: go mod tidy
Branch: `chore/go-mod-tidy`

Generated go.sum, pulled all dependencies. Straightforward.

### Task 2: TLS with SMP-Compatible CA Chain
Branch: `feat/smp-tls`

Implemented the full SMP TLS certificate architecture:
- Persistent Ed25519 CA certificate (saved to data/ca.key + data/ca.crt)
- Online server certificate signed by CA (rotated on each restart)
- X25519 DH key pair for handshake key exchange
- Forced TLS 1.2 with ECDHE_ECDSA_WITH_CHACHA20_POLY1305_SHA256
- ALPN "smp/1"

**Critical discovery:** Go's crypto/tls does not expose TLS Finished messages for TLS 1.3, which SMP requires for tls-unique channel binding (RFC 5929). The Haskell TLS library provides getPeerFinished for all versions. Solution: force TLS 1.2 where ConnectionState().PeerFinished is available.

**CA fingerprint:** SHA256 of the full DER-encoded certificate (cert.Raw), base64url without padding. This took three attempts to get right (see fix branches below).

### Task 3: SMP Version Handshake
Branch: `feat/smp-handshake`

Server sends ServerHello with version range (min=6, max=7), X25519 DH public key as SPKI, and session ID derived from tls-unique channel binding. Client responds with ClientHello containing agreed version, its own DH key, and CA fingerprint hash for verification.

### Task 4: PING/PONG Verification
Branch: `feat/ping-pong`

Integration test: full TLS connection, version handshake, then PING/PONG round-trip through 16 KB block framing. First end-to-end test of the complete connection pipeline.

### Task 5: NEW Command
Branch: `feat/queue-new`

Creates a queue with random 24-byte recipient and sender IDs. Parses Ed25519 auth key and X25519 DH key from SPKI-encoded body. Returns IDS response with both IDs and server's DH public key. Idempotent: NEW with same key returns existing IDS.

### Task 6: SUB Command
Branch: `feat/queue-sub`

Subscribe to a queue with Ed25519 signature verification. Signature covers sessionID + corrId + entityId + command tag. One subscriber per queue - new SUB sends END to displaced connection. Delivers first pending message immediately after subscribe.

### Task 7: KEY + SEND/MSG/ACK
Branch: `feat/message-delivery`

Complete message delivery cycle:
- KEY sets sender authentication key (recipient command, SPKI-encoded Ed25519)
- SEND verifies sender signature, stores message, delivers to active subscriber
- MSG delivers oldest unacknowledged message with timestamp and flags
- ACK acknowledges receipt, triggers cryptographic deletion, delivers next message
- DeliveryAttempts counter with MaxDeliveryAttempts=5 (redelivery loop protection)

### Task 8: Integration Tests
Branch: `feat/integration-tests`

13 integration tests covering full message flows:
- Queue creation and subscription
- Message send, receive, acknowledge
- Subscription takeover with END notification
- Error cases (auth failure, no queue, duplicate operations)

### Task 9: BadgerDB Store
Branch: `feat/badger-store`

Replaced MemoryStore with BadgerDB v4 for production:
- Per-message AES-256-GCM encryption with random 32-byte keys
- Cryptographic deletion: key zeroed before entry removal
- Native TTL: 48h default, 7d hard maximum
- GC every 5 minutes
- Lexicographic key ordering for FIFO message delivery
- QueueStore interface preserved - MemoryStore for tests, BadgerDB for production

## SimpleX Compatibility Fixes (8 Branches)

After Phase 1 was feature-complete, testing against the real SimpleX Chat desktop application revealed 8 protocol incompatibilities. Each was fixed on its own branch.

### Fix 1: CA Fingerprint - SPKI Hash
Branch: `fix/fingerprint-spki`

Changed fingerprint calculation to SHA256 of the SubjectPublicKeyInfo (SPKI) block instead of the full certificate DER. This was based on initial analysis of the SMP spec.

**Result:** Still rejected by SimpleX. Reverted in Fix 2.

### Fix 2: CA Fingerprint - Full Certificate DER
Branch: `fix/fingerprint-cert-raw`

Reverted fingerprint to SHA256 of the complete DER-encoded certificate (cert.Raw). Confirmed by analyzing the Haskell reference: `getFingerprint` uses `X.HashSHA256` on the full certificate, not the SPKI block.

**Lesson:** The SMP URI fingerprint = base64url_nopad(SHA256(cert.Raw)). Three different hash targets were tested (full cert, SPKI, raw public key) before confirming the correct one.

### Fix 3: Fingerprint Debug Logging
Branch: `fix/fingerprint-debug`

Added detailed hex logging of all three fingerprint variants (cert, SPKI, pubkey) to diagnose the handshake failure. This logging proved essential for all subsequent fixes.

### Fix 4: ClientHello keyHash Parsing
Branch: `fix/keyhash-parsing`

The ClientHello keyHash field was stored as a Go string but SimpleX sends raw 32-byte SHA256 hash bytes. Base64 decoding of raw bytes silently produced empty results. Fixed by changing KeyHash from string to []byte and using raw byte comparison with crypto/subtle.ConstantTimeCompare.

**Lesson:** When debug logging shows `client_keyhash_len=0`, the parsing layer is silently failing, not the comparison logic.

### Fix 5: IDS Response Debug Logging
Branch: `fix/ids-response-format`

Added hex dumps of NEW command body and IDS response to compare byte-by-byte with the expected SMP wire format. This revealed the padding issue (Fix 6) and the version-dependent format issue (Fix 7).

### Fix 6: Block Padding - '#' Not Zero
Branch: `fix/compare-real-server`

Created `cmd/smp-capture/main.go` diagnostic tool that connects to the official Haskell SMP server at smp.simplego.dev:5223 and captures raw 16 KB blocks for comparison.

**Discovery:** The SMP spec defines `pad = N*N"#"` - padding bytes are '#' (0x23), not zero bytes (0x00). GoRelay was using zero padding, causing SimpleX clients to reject every block as malformed.

Fixed PaddingByte from 0x00 to '#' and updated the padding loop in WriteBlock.

### Fix 7: IDS Response Version-Aware Format
Branch: `fix/ids-sndSecure`

The IDS response included a sndSecure boolean byte, but this field only exists in SMP v9+. SimpleX Chat negotiates v7, where sndSecure is not part of the IDS body. The extra byte caused CMD SYNTAX errors.

Fixed by making the response conditional on `client.smpVersion >= 9`. Also discovered that basicAuth in NEW is v9+ only.

### Fix 8: Error Format + SKEY Command
Branch: `fix/error-format-skey`

Two issues in one branch:

1. Error responses sent binary byte codes (e.g., 0x01 for AUTH) instead of text strings ("AUTH", "NO_QUEUE", "INTERNAL"). Created `errorCodeToText()` function mapping all internal error codes to their SMP wire text representation.

2. Added SKEY command handler for sender-side queue securing. After NEW + KEY, the sender calls SKEY with their Ed25519 key to secure the queue from their side. Required for SimpleX's `secureQueue` flow.

## Technical Discoveries

### TLS 1.2 vs 1.3 for SMP
Go's crypto/tls does not expose TLS Finished messages for TLS 1.3. SMP requires these for tls-unique channel binding (RFC 5929). The Haskell TLS library exposes getPeerFinished regardless of version. Workaround: force TLS 1.2 with ECDHE_ECDSA_WITH_CHACHA20_POLY1305_SHA256. Security equivalent for this use case.

### SPKI Encoding for Ed25519 and X25519
SMP encodes public keys as SubjectPublicKeyInfo (SPKI) DER structures, not raw 32-byte keys. SPKI adds a 12-byte ASN.1 header identifying the algorithm OID. Ed25519 SPKI is 44 bytes (12 header + 32 key). X25519 SPKI is 44 bytes (12 header + 32 key).

### shortString Encoding
SMP uses length-prefixed strings throughout: 1 byte length + N bytes data. This applies to queue IDs (0x18 + 24 bytes), public keys, signatures, and error messages.

### Text-Based Command Tags
SMP commands on the wire use text tags ("NEW ", "IDS ", "SUB", "KEY ", "SEND ", etc.), not binary command codes. The space after some tags is significant - it separates the tag from the body.

### Redelivery Loop Attack
Evgeny Poberezkin (SimpleX inventor) described an attack where a crafted message crashes the client before ACK, causing redelivery for the entire TTL period. On official servers with 21-day TTL, this bricks the device for three weeks. GoRelay's MaxDeliveryAttempts=5 combined with 48h TTL limits exposure to minutes.

| ADR-009 | SMP transport | TLS 1.2 | Go lacks TLS 1.3 Finished message API for tls-unique |

## Lessons Learned

1. **Test against real clients early.** Unit tests passed but SimpleX Chat rejected everything. Eight compatibility fixes were needed after "feature complete."

2. **Wire format details matter enormously.** A single wrong byte (0x00 vs 0x23 for padding) causes silent rejection. Hex dump logging is not optional.

3. **Version-dependent fields are a minefield.** SMP v7 and v9 have different IDS response bodies. The spec documents the latest version; older versions require reading the Haskell source.

4. **Text-based protocols are deceptively complex.** "ERR AUTH" looks simple but the encoding (text tag + space + text error code) has multiple points of failure.

5. **The SMP spec is incomplete without the Haskell source.** The protocol document describes intent; the server source code is the ground truth for wire format details.
