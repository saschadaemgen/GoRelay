# Season 003 - Two Clients Chat Over GoRelay

**Date:** 2026-03-22 to 2026-03-23
**Duration:** One overnight session (approximately 20 hours elapsed, two working blocks)
**Participants:** Sascha Daemgen (the Prince), Mausi (Architecture/Strategy), Hasi (Claude Code)
**Status:** Complete

---

## Overview

Season 3 achieved the primary goal of GoRelay Phase 2: two real SimpleX Chat clients exchanging encrypted messages through GoRelay. This required implementing NaCl crypto_box encryption for MSG delivery (Layer 3 server-to-recipient encryption), fixing the rcvMsgBody wire format to match the Haskell reference implementation byte-for-byte, and ultimately discovering that the SimpleX server uses standard NaCl secretbox rather than a custom XSalsa20 variant.

The session was characterized by deep protocol debugging. Fourteen deployments were made to the production VPS. Multiple theories about the encryption format were tested and discarded. The breakthrough came from consulting two independent authoritative sources (the Haskell simplexmq source code and the SimpleGo ESP32 project documentation) which jointly confirmed the exact byte-level format and crypto approach. The final fix was remarkably simple: replacing 101 lines of hand-rolled XSalsa20 code with 23 lines using Go's standard NaCl secretbox library.

At 12:40 CET on March 23, 2026, Bob's message "Hallo ueber GoRelay!" was delivered to Alice through the GoRelay server. GoRelay became the first known third-party SMP-compatible relay server written in Go.

## Block 1: NaCl crypto_box Implementation (March 22, Evening)

### Initial Implementation

The session began with formulating the Claude Code task for NaCl crypto_box MSG delivery. The SMP protocol requires the server to re-encrypt message bodies before delivery using a DH-derived shared secret between the server and the recipient.

The initial implementation created `internal/protocol/smp/cryptobox.go` with an `EncryptMsgBody()` function. The approach used `box.Precompute` (NaCl's `crypto_box_beforenm`) for the DH shared secret derivation, followed by `secretbox.Seal` for encryption.

Four MSG delivery points were identified and instrumented: two in `handleSUB` (immediate delivery on subscribe), one in `handleSEND` (immediate delivery to active subscriber), and one in `handleACK` (next message delivery after acknowledgment).

The `recipientDhPublicKey` from the NEW command body had previously been skipped during parsing. It was now extracted and used together with the server's DH private key to compute the shared secret. An `UpdateQueueDH()` method was added to the store interface for DH key persistence.

### Version Range Experiment

The SimpleX Desktop client initially reported "BROKER TRANSPORT version" errors. The version range was temporarily raised from v6-v7 to v6-v18 to match the official Haskell server's advertised range. This caused "block too short" errors because the client sent v14+ format data that GoRelay could not parse. The version range was reverted to v6-v7, which is what GoRelay actually implements. The version error turned out to be caused by Private Routing (PRXY), not version mismatch.

### PRXY Command Handling

The SimpleX client attempted to use Private Routing (PRXY command), which routes messages through a proxy server to hide the sender's connection to the destination server. GoRelay does not support proxy routing. A handler was added to return `ERR CMD PROHIBITED`, causing the client to fall back to direct connection.

### Unsigned SEND for Confirmation

The first SEND on a new queue is the SMP Confirmation Message, which is sent before the sender has been authenticated with KEY. The original handleSEND required a signature on every SEND. The fix allowed unsigned SEND when `SenderKey == nil`, permitting the confirmation flow to complete.

### Shutdown Timeout

The server previously hung on Ctrl+C because `clientWg.Wait()` blocked indefinitely when connections were still open. A 10-second timeout was added to the shutdown function, allowing clean server restarts during debugging.

### CLI Server Test: PASSED

Using the SimpleX Chat CLI installed in WSL (`simplex-chat`), the `/smp test` command against GoRelay returned success. This verified that the SMP handshake, NEW, KEY, SEND, SUB, MSG, ACK, and DEL commands all worked correctly at the protocol level.

### First Connection Attempt

Alice created a contact address (`/ad`) and Bob attempted to connect (`/c <link>`). Bob's confirmation message was sent and acknowledged. The server delivered a 16,141-byte MSG to Alice. Alice's client sent an ACK. But Alice showed no contact request.

**Diagnosis:** The SMP transport layer worked perfectly. Messages flowed end-to-end. But Alice's client could not decrypt the MSG body, causing the agent layer to silently discard the message.

## Block 2: Custom XSalsa20 - The Wrong Path (March 22, Night)

### The SimpleGo Warning

The SimpleGo ESP32 project documentation contained an explicit warning: "rcvDhSecret = crypto_scalarmult (raw X25519 DH output). NOT crypto_box_beforenm - that would do an additional HSalsa20 derivation that SimpleX doesn't want here."

Based on this, the `box.Precompute` call was replaced with raw `ecdh.X25519().ECDH()` output. The raw 32-byte ECDH result was stored directly as `ServerDHSecret`. Commit `46913fd`.

### Private Routing Workaround

Bob's SimpleX client attempted to connect through `smp5.simplex.im` as a proxy server, which rejected the connection with a version error. The fix was to disable Private Routing in Bob's client: `/network socks=off smp-proxy=never smp-proxy-fallback=yes`. After this, Bob connected directly to GoRelay.

### The Custom XSalsa20 Theory

SimpleGo documentation described a non-standard XSalsa20 nonce splitting used by Haskell's cryptonite library:

```
Standard NaCl:  HSalsa20(key, nonce[0:16]),  Salsa20(subkey, nonce[16:24])
SimpleX custom: HSalsa20(key, nonce[8:24]),  Salsa20(subkey, nonce[0:8])
```

A custom implementation was written using `poly1305`, `salsa20`, and `salsa20/salsa` packages, replacing the standard `nacl/secretbox`. The `simplexCryptoBox()` and `SimplexCryptoBoxOpen()` functions were implemented from scratch. Commit `9fc7c36`.

### Three-Step HSalsa20

Further investigation of the cryptonite library revealed that the initialization does HSalsa20 with a zero nonce before the main derivation. This added a third step:

```
Step 1: HSalsa20(key, zeros[16])      -> subkey1  (cryptonite initialize)
Step 2: HSalsa20(subkey1, nonce[8:24]) -> subkey2  (cryptonite derive)
Step 3: Salsa20(subkey2, nonce[0:8])   -> keystream
```

The implementation was updated to include all three steps. Commit `189ec1f`.

### Test Vector Verification

Real test vectors from SimpleGo ESP32 debugging sessions were available:

```
DH Secret:  2fa438b42c75071d4d68729f7bf8977ca30dada823a59c44119c9ec9d68fbf2d
Nonce:      6ea7fa47cabc1eb2ae550c6b0e73d539326b30b0a29c8a0d
Expected subkey1: ce1b436c8b333a5ff881d4c09c55545594f1b73a214ea61c534936cd0d397912
```

A test was written to verify Step 1. Result: **Step 1 PASSED** - `HSalsa20(key, zeros)` produced the exact expected subkey. Step 2 subkey also matched when verified by the SimpleGo team. However, full decrypt of the test vector ciphertext failed because the test vector contained only 64 bytes of ciphertext (truncated), making MAC verification impossible. The partial test confirmed the HSalsa20 implementation was correct but could not prove end-to-end crypto correctness.

## Block 3: Wire Format Debugging (March 23, Morning)

With the crypto implementation seemingly correct (test vectors matched for Steps 1 and 2), attention turned to the wire format of `rcvMsgBody` - the plaintext structure that gets encrypted before MSG delivery.

### Diagnostic Logging

Temporary INFO-level logging with "DIAG:" prefix was added at every stage of the encryption pipeline:
- DH shared secret (first 16 bytes) and length
- msgId/nonce (full hex)
- sentBody (first 32 bytes and length)
- rcvMsgBody length after assembly
- Padded plaintext (first 32 bytes and length)
- Encrypted output (first 32 bytes and length)

These logs proved essential for the next eight iterations.

### Iteration 1: Duplicate Flags and SP (Commit b87d249)

**Theory:** The sentBody from SEND already contained `flags + SP + smpEncMessage`. The code was adding flags and SP again, creating 2 extra bytes.

**Fix:** Changed from `rcvMsgBody = timestamp + flags + SP + sentBody` to `rcvMsgBody = timestamp + sentBody`.

**Result:** `rcvMsgBody_len` changed from 16004 to 16002. Still did not work.

**Diagnosis from DIAG logs:** The sentBody started with `54 20` (ASCII "T" + space), confirming that flags and SP were already in the sentBody. But the code was now passing them through raw instead of converting to binary format.

### Iteration 2: ASCII T/F Flags (Commit 0e43fe0)

**Theory:** The flags byte needed to be binary (0x01/0x00) instead of ASCII ('T'/'F').

**Fix:** Changed `flagsByte = 0x01` to `flagsByte = 'T'` (0x54) and `0x00` to `'F'` (0x46).

**Result:** Still did not work. Later confirmed that ASCII encoding was correct per the Haskell spec.

### Iteration 3: uint16BE Length Prefix (Commit 04897ae)

**Theory:** Haskell's `smpEncode` for ByteString adds a uint16BE length prefix before the message body inside rcvMsgBody.

**Fix:** Added 2-byte `binary.BigEndian.PutUint16` length prefix before smpEncMessage.

**Result:** `rcvMsgBody_len` changed to 16003. Still did not work.

### Iteration 4: 12-Byte Timestamp (Commit 371d994)

**Theory:** Haskell's SystemTime includes nanoseconds, making the timestamp 12 bytes (8 seconds + 4 nanoseconds) instead of 8.

**Fix:** Changed timestamp from 8 bytes to 12 bytes (`int64BE seconds + uint32BE nanoseconds`).

**Result:** `rcvMsgBody_len` changed to 16007. Still did not work.

### Consultation: Haskell Source and SimpleGo

At this point, having made four format changes without success, the decision was made to stop guessing and consult authoritative sources.

**Source 1: Haskell simplexmq repository.** Claude Code cloned the official repository and analyzed `Protocol.hs`, `Encoding.hs`, and `Crypto.hs`. Key findings:

- `SystemTime` encodes as 8 bytes Int64 BE (nanoseconds are discarded)
- `MsgFlags` encodes as 1 byte ASCII ('T' or 'F')
- `ByteString` uses 1-byte length prefix, `Tail` uses no prefix (raw bytes)
- `rcvMsgBody = smpEncode(msgTs) ++ smpEncode(msgFlags) ++ ' ' ++ msgBody`
- `padMaxLenBS` adds uint16BE length prefix and pads with '#' to `MaxRcvMessageLen`
- `MaxRcvMessageLen = 16104`, padded size = 16106
- MSG wire format: `"MSG " + shortString(msgId) + Tail(encryptedBody)`

**Source 2: SimpleGo ESP32 project.** The SimpleGo team provided byte-level offset tables from verified hex dumps:

```
rcvMsgBody after decrypt + unpad:
[0-7]   timestamp    8 bytes Int64 BE
[8]     msgFlags     1 byte ASCII 'T' or 'F'
[9]     space        1 byte 0x20
[10+]   sentMsgBody  Raw bytes (= what sender sent via SEND)
```

Both sources agreed completely. This revealed five simultaneous errors in the current code.

### Iteration 5: Five-Fix Commit (Commit 4c974f8)

Based on the confirmed specification from both sources:

1. **MaxMessageLength:** Changed from 16064 to 16104. paddedSize from 16066 to 16106.
2. **Timestamp:** Reverted from 12 bytes back to 8 bytes Int64 BE.
3. **rcvMsgBody format:** Changed to `timestamp(8) + sentBody(raw)`. Removed all flag-parsing code. The sentBody already contains `flags + SP + smpEncMessage` in the correct format.
4. **Padding character:** Changed from zero bytes (0x00) to '#' (0x23) per Haskell `B.replicate padLen '#'`.
5. **MSG wire format:** Confirmed correct - no cleartext timestamp/flags between msgId and encrypted body.

**Result after deployment:**

```
rcvMsgBody_len: 16002  (correct: 8 + 15994)
padded_len:     16106  (correct)
encrypted_len:  16122  (correct: 16 + 16106)
payload_len:    16181  (correct)
```

All sizes matched the Haskell reference. Still did not work.

### Self-Decrypt Verification (Commit f467f3f)

A self-decrypt check was added: immediately after encrypting the MSG body, the server attempted to decrypt its own output.

**Result:** Self-decrypt succeeded. The decrypted first 16 bytes matched the padded plaintext exactly. This proved that the format was correct and that our encrypt/decrypt was internally consistent. The problem was that our crypto produced different ciphertext than what the Haskell client expected.

## Block 4: The Crypto Breakthrough (March 23, Late Morning)

### The Comparison Test (Commit 7642f35)

A test was written comparing three crypto approaches with identical inputs:

| Approach | First 8 bytes |
|----------|---------------|
| Custom 3-step HSalsa20 | `d4c997d425be105e` |
| Standard NaCl (beforenm + secretbox) | `9d0798bd2f8dc4df` |
| Direct secretbox (raw key) | `978ae5d9d96621f4` |

All three produced different output. The custom implementation (what we had been using) did not match standard NaCl.

### The Root Cause

The Haskell SMP server uses `cbEncrypt` which internally calls `crypto_box_afternm`. This is equivalent to:

1. `crypto_box_beforenm(raw_dh_key)` = `HSalsa20(key, zeros[16])` -> shared key
2. `crypto_secretbox(shared_key, nonce, plaintext)` = standard XSalsa20-Poly1305

Standard NaCl's `crypto_secretbox` (XSalsa20) uses this nonce splitting:
```
HSalsa20(key, nonce[0:16])  -> subkey
Salsa20(subkey, nonce[16:24]) -> keystream
```

Our custom implementation (from SimpleGo documentation) used a DIFFERENT nonce splitting:
```
HSalsa20(key, nonce[8:24])  -> subkey
Salsa20(subkey, nonce[0:8])  -> keystream
```

The SimpleGo custom variant is used for **Layer 2 (client-to-client E2E encryption)** where the Haskell cryptonite library is used directly. But the SMP server uses **Layer 3 (server-to-recipient encryption)** which uses standard NaCl. The two layers use different crypto libraries with different nonce splitting conventions. This distinction was the root cause of the entire debugging session.

### The Final Fix

The `simplexCryptoBox` and `SimplexCryptoBoxOpen` functions were replaced entirely:

```go
func simplexCryptoBox(key [32]byte, nonce [24]byte, plaintext []byte) []byte {
    var beforenmKey [32]byte
    var zeros16 [16]byte
    salsa.HSalsa20(&beforenmKey, &zeros16, &key, &salsa.Sigma)
    result := secretbox.Seal(nil, plaintext, &nonce, &beforenmKey)
    for i := range beforenmKey { beforenmKey[i] = 0 }
    return result
}
```

Net change: -101 lines, +23 lines. The custom Poly1305 handling, manual Salsa20 keystream generation, and hand-rolled XOR operations were all replaced by a single `secretbox.Seal` call.

### Success

Deployment at 12:39 CET. Alice created a contact address. Bob connected with Private Routing disabled. The server logs showed:

## Technical Discoveries

### Layer 2 vs Layer 3 Crypto

The most significant discovery of this session: SimpleX uses two DIFFERENT crypto implementations for two different encryption layers.

**Layer 3 (server-to-recipient):** Standard NaCl `crypto_box` / `crypto_secretbox`. Uses `HSalsa20(key, zeros)` as `crypto_box_beforenm` for key derivation, then standard XSalsa20-Poly1305 with `nonce[0:16]` / `nonce[16:24]` splitting. This is what the SMP server uses.

**Layer 2 (client-to-client E2E):** Haskell cryptonite library with non-standard nonce splitting. Uses `HSalsa20(key, zeros)` for initialization, then `HSalsa20(subkey, nonce[8:24])` for derivation, then `Salsa20(subkey2, nonce[0:8])` for keystream. This is what SimpleGo implements for ESP32 client-side crypto.

Confusing one for the other caused 12+ hours of debugging. The SimpleGo documentation warned about `crypto_box_beforenm` but in the context of Layer 2 key derivation (raw DH vs precomputed), not about the nonce splitting difference. Both layers share Step 1 (`HSalsa20(key, zeros)`), which is why the test vector for Step 1 passed while the full encryption failed.

### rcvMsgBody Confirmed Format

```
[0-7]    timestamp     8 bytes Int64 BE (Unix epoch seconds)
[8]      msgFlags      1 byte ASCII 'T' (notification) or 'F'
[9]      space         1 byte 0x20
[10+]    sentMsgBody   Raw bytes from SEND command body
```

Padded as: `uint16BE(len(rcvMsgBody)) + rcvMsgBody + '#' padding to 16106 bytes`

Encrypted as: `NaCl secretbox(beforenm_key, msgId_as_nonce, padded)` producing `16 byte MAC + 16106 byte ciphertext = 16122 bytes`

### MSG Wire Format

```
"MSG " + shortString(msgId) + encryptedBody(Tail)
```

No cleartext timestamp or flags between msgId and encrypted body. The timestamp and flags are inside the encrypted payload. `Tail` encoding means raw bytes with no length prefix - the encrypted body consumes all remaining bytes in the transmission.

### MaxRcvMessageLen

The SMP spec states `maxMessageLength = 16064` but the actual maximum for `rcvMsgBody` (after adding timestamp, flags, and space) is `MaxRcvMessageLen = 16104`. The padded size is `16104 + 2 = 16106`. The encrypted size is `16106 + 16 = 16122`. The SMP spec's 16064 refers to the maximum `sentMsgBody` size, not the maximum `rcvMsgBody` size.

### Padding Character

The padding character for `padMaxLenBS` is '#' (0x23), consistent with the block-level padding discovered in Season 2. The SimpleX protocol uses '#' as the universal padding byte at all levels.

### Private Routing (PRXY)

SimpleX Chat v5.8+ uses Private Routing by default for "unknown" servers. When connecting to a new SMP server, the client attempts to route through a known proxy server (one of the official SimpleX servers). The proxy connects to the destination server on the client's behalf. GoRelay does not support being a proxy target. The workaround is to disable Private Routing on the client: `/network socks=off smp-proxy=never smp-proxy-fallback=yes`.

## Theories Tested and Discarded

The following theories were investigated during debugging and found to be incorrect or incomplete:

1. **box.Precompute is wrong:** Partially correct. The raw DH output is needed, but box.Precompute does the same thing as our Step 1 (HSalsa20 with zeros). The real issue was Steps 2-3 nonce splitting.

2. **SimpleX uses custom XSalsa20 nonce splitting:** True for Layer 2 (client-to-client), false for Layer 3 (server-to-recipient). The confusion between layers caused the majority of debugging time.

3. **Duplicate flags+SP in rcvMsgBody:** Was a real bug initially (sentBody already had flags), but fixing it alone was insufficient.

4. **Flags must be binary 0x01/0x00:** Incorrect. SMP uses ASCII 'T'/'F' for all boolean encoding.

5. **uint16BE length prefix before smpEncMessage:** Incorrect. Haskell uses `Tail` encoding (raw bytes, no prefix) for the last field. The uint16BE prefix is only in the outer padding layer.

6. **Timestamp is 12 bytes (seconds + nanoseconds):** Incorrect. Haskell's smpEncode for SystemTime discards nanoseconds: `smpEncode = smpEncode . systemSeconds`.

7. **Cleartext timestamp+flags in MSG wire format:** Incorrect. These fields are inside the encrypted body, not cleartext between msgId and encrypted body.

8. **Padding is zero bytes:** Incorrect (also caught in Season 2). All padding in SMP is '#' (0x23).

## Lessons Learned

1. **Consult the source, not the documentation, not the secondary sources.** The SMP specification says `maxMessageLength = 16064` but the Haskell implementation uses `MaxRcvMessageLen = 16104`. The SimpleGo documentation described Layer 2 crypto when we needed Layer 3. Only reading the actual Haskell source (`Protocol.hs`, `Encoding.hs`, `Crypto.hs`) and cross-referencing with a second working implementation (SimpleGo) gave the definitive answer.

2. **Two different crypto approaches for two different layers is a trap.** SimpleX uses Haskell cryptonite (custom nonce splitting) for client-side E2E and standard NaCl (standard nonce splitting) for server-side message encryption. Both share the first derivation step, which makes partial test vectors misleading - Step 1 passes but the full encryption produces different output.

3. **Self-decrypt verification is a powerful diagnostic.** When self-decrypt passes but the client cannot decrypt, the problem is definitively in the crypto approach (algorithm choice), not the format or key material. This narrows the search space enormously.

4. **Comparison tests between crypto approaches are decisive.** The test comparing custom vs standard NaCl output immediately showed they produce different results. This 10-line test saved hours of further format debugging.

5. **Less code is often the fix.** The final solution was -101 lines, +23 lines. The hand-rolled XSalsa20 implementation with manual Poly1305, keystream generation, and XOR operations was replaced by `secretbox.Seal` - a single, audited, well-tested library call. The custom implementation was correct for its intended purpose (Layer 2) but wrong for the actual use case (Layer 3).

6. **Fourteen deployments is not failure, it is iteration.** Each deployment tested a specific hypothesis and produced concrete diagnostic data. The DIAG logging framework made each iteration fast: deploy, connect, read logs, identify next hypothesis.

7. **Wire format and crypto are separate failure modes.** The session cleanly separated into format debugging (Iterations 1-5, confirmed by matching the Haskell reference sizes) and crypto debugging (comparison test, secretbox replacement). Conflating the two would have made the problem unsolvable.
